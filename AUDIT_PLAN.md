## Codebase Context (read before executing any phase)

**What this app is**: A DM (Dungeon Master) screen for tabletop RPG. Two browser views run simultaneously â€” the DM view (`/`) and the Player view (`/players/:slug`). A Prep mode (`/prep`) handles all data management. Communication between DM and players goes through PartyKit WebSockets via `src/lib/sync.ts`.

**State**: 9 Zustand stores in `src/store/`, all using `persist` middleware to localStorage. Each store exports its types, a `STORE_KEY` constant (most of them), and a `migrateXxxStore` function that is imported and tested in `src/store/__tests__/migrations.test.ts`.

**Key architectural rules**:

- Nothing touches `BroadcastChannel` or `PartySocket` directly â€” everything goes through `src/lib/sync.ts`
- `sendMessage()` in sync.ts strips `hp`/`maxHp` before broadcast (DM-only data)
- The migration test file imports all `migrateXxx` functions by name â€” if you rename one, update the test import too
- Run `npm run format` before staging (pre-commit hook does NOT re-stage formatted files)
- Run `npm test` to verify (Vitest, runs in happy-dom)

**Component naming**: Components in `src/components/ui/` are primitives (shadcn/ui wrappers). Everything else is feature-specific. The `src/components/characters/initiative/` folder owns the initiative tracker sub-system.

### Bug 1A: `InitiativeSetupDialog` shows stale character list on reopen

**File**: `src/components/characters/initiative/initiativeSetupDialog.tsx`

**Problem**: Line 50 initializes actors with a `useState` lazy initializer:

```ts
const [actors, setActors] = useState<Actor[]>(() => newPlayerActors(characters));
```

This only runs once when the component first mounts. If a character is added in Prep Mode while this component is already mounted (but the dialog is closed), reopening the dialog still shows the old list. The new character won't appear until a full page reload.

**Fix**: Add a `useEffect` that resets actors to the current character list whenever `isOpen` transitions to `true`. Place it after the existing `useState` declarations (around line 51):

```ts
useEffect(() => {
  if (isOpen) setActors(newPlayerActors(characters));
}, [isOpen]); // eslint-disable-line react-hooks/exhaustive-deps
```

The exhaustive-deps lint suppression is intentional: `characters` and `newPlayerActors` should NOT be in the dep array â€” we only want to re-seed when the dialog opens, not on every character change while it's open (that would discard in-progress initiative rolls).

**Manual test**:

1. Open the DM screen. Go to Prep â†’ Player Characters, add a new character named "TestChar".
2. Go back to DM Screen â†’ Combat tab. Click "New Combat".
3. Verify "TestChar" appears in the Players list.

### Bug 1B: Open5e search has a fetch race condition

**File**: `src/components/encounters/open5eSearchDialog.tsx`

**Problem**: Lines 186â€“208. The debounced `useEffect` fires a `fetch()` with no `AbortController`. If the user types "gob" then quickly types "goblin", two requests go out. The "gob" response (slower, larger result set) can resolve after "goblin" and overwrite the correct results. Also, if the dialog is closed while a fetch is in-flight, calling `setResults`/`setLoading` on the unmounted component produces a React warning.

**Fix**: Introduce an `AbortController` inside the `useEffect`. The existing `debounceRef` pattern stays â€” just add abort signaling to the fetch:

Replace the `useEffect` body (lines 183â€“208) with:

```ts
useEffect(() => {
  if (debounceRef.current) clearTimeout(debounceRef.current);
  if (query.trim().length < 2) {
    setResults([]);
    return;
  }
  const controller = new AbortController();
  debounceRef.current = setTimeout(async () => {
    setLoading(true);
    setError(null);
    try {
      const res = await fetch(
        `https://api.open5e.com/v2/creatures/?name__icontains=${encodeURIComponent(
          query.trim()
        )}&limit=20`,
        { signal: controller.signal }
      );
      if (!res.ok) throw new Error('Request failed');
      const data = await res.json();
      setResults(data.results ?? []);
    } catch (err) {
      if ((err as Error).name === 'AbortError') return;
      setError('Could not reach Open5e. Check your connection.');
      setResults([]);
    } finally {
      setLoading(false);
    }
  }, 350);
  return () => {
    if (debounceRef.current) clearTimeout(debounceRef.current);
    controller.abort();
  };
}, [query]);
```

**Manual test**:

1. Go to Prep â†’ NPCs â†’ "Add from Open5e".
2. Type "gob" very quickly followed by "goblin" â€” results should consistently show "goblin" matches, never "gob" matches.
3. Open the dialog, start typing, then close the dialog immediately â€” no console errors about updating an unmounted component.

### 2A: Extract `useScaleToFit` hook from `player-view.tsx`

**File**: `src/pages/player-view.tsx` (read lines 48â€“83 for full context)

**Problem**: Two `useEffect` blocks (lines 56â€“69 and lines 71â€“83) are identical in logic. Both observe an element's height and compute `scale = Math.min(1, (window.innerHeight * 0.9) / el.offsetHeight)`. The only difference is which `ref` and which `setState` they use.

**Fix â€” Step 1**: Create `src/lib/useScaleToFit.ts`:

```ts
import { useEffect, useState } from 'react';

export function useScaleToFit(ref: React.RefObject<HTMLElement | null>, dep: unknown): number {
  const [scale, setScale] = useState(1);

  useEffect(() => {
    const el = ref.current;
    if (!el) return;
    const update = () => setScale(Math.min(1, (window.innerHeight * 0.9) / el.offsetHeight));
    update();
    const ro = new ResizeObserver(update);
    ro.observe(el);
    window.addEventListener('resize', update);
    return () => {
      ro.disconnect();
      window.removeEventListener('resize', update);
    };
  }, [dep]); // eslint-disable-line react-hooks/exhaustive-deps

  return scale;
}
```

The `dep` parameter re-triggers the effect when the observed element's _visibility_ changes (in `player-view.tsx` both effects depend on `imageSource`). The ref itself is stable so it doesn't need to be a dep.

**Fix â€” Step 2**: In `player-view.tsx`, replace the two `useEffect` blocks and the four state/ref declarations with:

```ts
// Remove these 4 lines:
// const [centeredScale, setCenteredScale] = useState(1);
// const [cornerScale, setCornerScale] = useState(1);
// (the two useEffect blocks for scaling)

// Add at the top (with other imports):
import { useScaleToFit } from '../lib/useScaleToFit';

// Replace the 4 declarations + 2 effects with:
const centeredScale = useScaleToFit(centeredContentRef, imageSource);
const cornerScale = useScaleToFit(cornerContentRef, imageSource);
```

The `centeredContentRef` and `cornerContentRef` refs stay as-is.

### 2B: Extract `<ImportButton />` from both page headers

**Files**: `src/pages/dm-screen.tsx` (lines 24, 107â€“117) and `src/pages/prep-screen.tsx` (lines 17, 56â€“67)

**Problem**: Both pages have an identical hidden-file-input pattern:

- `const fileInputRef = useRef<HTMLInputElement>(null)`
- An `<input type="file" accept=".json" className="hidden" ref={fileInputRef}>`
- A trigger button's `onClick` that calls `fileInputRef.current?.click()`
- An `onChange` handler that calls `importData(file)` and resets the input

**Fix â€” Step 1**: Create `src/components/ui/import-button.tsx`:

```tsx
import { useRef } from 'react';
import { Download } from 'lucide-react';
import { Button } from '@/components/ui/button';
import { importData } from '@/lib/exportImport';

interface ImportButtonProps {
  variant?: React.ComponentProps<typeof Button>['variant'];
  size?: React.ComponentProps<typeof Button>['size'];
  className?: string;
  children?: React.ReactNode;
}

export default function ImportButton({
  variant = 'outline',
  size = 'sm',
  className,
  children
}: ImportButtonProps) {
  const fileInputRef = useRef<HTMLInputElement>(null);

  return (
    <>
      <Button
        variant={variant}
        size={size}
        className={className}
        onClick={() => fileInputRef.current?.click()}>
        {children ?? (
          <>
            <Download className='h-4 w-4' />
            Import
          </>
        )}
      </Button>
      <input
        ref={fileInputRef}
        type='file'
        accept='.json'
        className='hidden'
        onChange={(e) => {
          const file = e.target.files?.[0];
          if (file) importData(file);
          e.target.value = '';
        }}
      />
    </>
  );
}
```

**Fix â€” Step 2**: In `dm-screen.tsx`:

- Remove `fileInputRef` declaration and the hidden `<input>` element
- Remove the `importData` import if it becomes unused after removing the inline handler (keep `exportData`)
- Replace the Import `DropdownMenuItem` trigger with `<ImportButton>` (wrap in a `DropdownMenuItem` using `asChild` or restructure as needed to fit the dropdown)

Note: In `dm-screen.tsx` the import trigger is a `DropdownMenuItem`, not a standalone button. The `ImportButton` component's trigger button can be replaced or the component can accept a `asChild`-style render prop. Simplest approach: keep the `DropdownMenuItem` wrapper but replace its `onSelect` body with the click-trigger pattern extracted into `ImportButton` â€” or just inline the file input as a sibling to the dropdown and have the `DropdownMenuItem` call `fileInputRef.current?.click()` on the shared ref. Use whichever approach keeps the visual appearance unchanged.

### 2D: Consolidate `toUrlSlug` / `nameToSlug` duplication

**Files**: `src/components/campaigns/campaignEditDialog.tsx` (lines 23â€“29) and `src/store/campaignStore.ts` (lines 25â€“33)

**Problem**: `campaignEditDialog.tsx` defines a local `toUrlSlug` function that is almost identical to `nameToSlug` in `campaignStore.ts`. The store version appends a 4-char UUID suffix; the dialog version doesn't. Both do the same lowercase/clean/trim/hyphenate transformation.

**Fix**: In `campaignStore.ts`, extract the base transformation into an exported helper and call it from `nameToSlug`:

```ts
// New export â€” base slug without the collision-prevention suffix
export function toBaseSlug(name: string): string {
  return name
    .toLowerCase()
    .replace(/[^a-z0-9\s-]/g, '')
    .trim()
    .replace(/\s+/g, '-');
}

// Existing function, now calls the shared base
export function nameToSlug(name: string, id: string): string {
  const suffix = id.replace(/-/g, '').slice(0, 4);
  return `${toBaseSlug(name)}-${suffix}`;
}
```

In `campaignEditDialog.tsx`:

- Delete the local `toUrlSlug` function
- Import `toBaseSlug` from `'../../store/campaignStore'`
- Replace all calls to `toUrlSlug(value)` with `toBaseSlug(value)`
